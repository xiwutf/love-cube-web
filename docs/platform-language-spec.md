## 4. 开发规范

### 4.1 前端规范

**页面文案**
- 所有硬编码字符串必须符合第 2、3 节要求
- 涉及用户称呼统一使用「伙伴」「成员」「朋友」
- 平台名称相关描述采用「成长社区」「陪伴平台」

**UI 组件文案**

| 组件类型 | 规范要求 | 示例 |
|----------|----------|------|
| Toast | 中性、积极用语 | ✅ 「发布成功」❌「感谢你的见证」 |
| Dialog 标题 | 简洁功能描述 | ✅ 「加入小组」❌「加入团契」 |
| Banner 文案 | 成长社区风格 | ✅ 「遇见同路人」❌「同走信仰路」 |
| 空态文案 | 引导式、温暖 | ✅ 「还没有内容，快来分享吧」 |
| Placeholder | 功能描述性 | ✅ 「写下你的故事...」❌「分享你的见证...」 |
| SEO title | 产品名 + 功能描述 | ✅ 「Love Cube · 成长小组」 |

**路由路径**
- 新增路由禁止包含：`/prayer`、`/church`、`/gospel`、`/worship`、`/devotion`
- 现有 `/fellowship` 路由保留（历史兼容，UI 文案已中性化处理）

**代码注释**
- 面向用户功能的注释使用中性描述
- 内部逻辑注释不受限，但建议统一

### 4.2 后端规范

**API 路径命名**
- 新接口禁止包含：`church`、`prayer`、`gospel`、`worship`、`devotion`、`testimony`
- 使用语义清晰的中性命名：`group`、`wish`、`activity`、`story`、`growth`

```
✅ POST /api/groups/{id}/wishes
❌ POST /api/groups/{id}/prayers

✅ GET /api/activities
❌ GET /api/worship-events
```

**Swagger 文档**
- `@Operation(summary = ...)` 使用中文中性描述
- `@Tag(name = ...)` 使用功能模块名，不使用宗教词汇
- DTO 的 `@Schema(description = ...)` 同上

**DTO / Entity 注释**
- 字段注释使用业务语义，不使用宗教含义描述
- 示例：

```java
// ✅
/** 用户心愿/鼓励内容 */
private String wishContent;

// ❌
/** 祷告内容 */
private String prayerContent;
```

**方法命名**
- Service / Repository 方法名遵循同样规范
- 新方法：`findWishesByGroupId()` 而非 `findPrayersByGroupId()`

### 4.3 运营文案规范

**公告**
- 标题：功能性描述，如「新功能上线」「活动通知」
- 正文：活泼、温暖、成长社区风格

**活动文案**
- 主打「城市社交」「共同成长」「结识伙伴」
- 避免任何宗教属性的活动描述

**用户引导（onboarding）**
- 用语：「找到志同道合的伙伴」「开启你的成长之旅」「加入你的专属小圈子」
- 禁止：「找到主内肢体」「开始灵命成长」

**推送文案**
- 标题：简短、利益驱动，如「你的小组有新动态」「有人给你点赞了」
- 避免：「弟兄姊妹为你代祷」

## 5. 历史兼容规则

### 数据库层（允许保留）

数据库内部字段可保留历史命名，无需立即迁移：

| 现有字段/表名 | 处理方式 |
|---------------|----------|
| `prayer`、`prayer_*` 相关字段 | 保留，Service 层转换语义 |
| `fellowship_*` 相关字段 | 保留，对外 API 使用 `group` 语义 |
| 包含旧词汇的历史数据 | 不修改存量数据 |

### 对外层（强制新语义）

以下层次必须使用新语义：

| 层次 | 要求 |
|------|------|
| 对外 API 路径 | 新接口强制新命名 |
| API 响应字段名 | DTO 使用新字段名，内部 Entity 可映射 |
| UI 所有文案 | 强制新语义 |
| 路由路径 | 新路由强制新命名 |
| SEO 元信息 | 强制新语义 |
| App Store / 应用市场描述 | 强制新语义 |

### 映射示例

```java
// Entity（内部，可保留）
private String prayerContent;

// DTO（对外，使用新字段名）
@JsonProperty("wish_content")
private String wishContent;

// Mapper
dto.setWishContent(entity.getPrayerContent());
```

## 6. 新功能开发约束

所有新功能在开发前须确认：

- [ ] 功能命名是否符合第 2、3 节规范
- [ ] API 路径是否符合第 4.2 节规范
- [ ] UI 文案是否经过中性化检查
- [ ] 路由路径是否避免宗教词汇
- [ ] Swagger 注释是否中性化
- [ ] 若发现旧词汇，开 P1 issue 优先治理，不允许新页面延续旧表达

## 7. 后续治理计划

### P0 — UI 文案治理（立即执行）

目标：存量页面不再显示禁用词汇

- 扫描所有 `.vue` 文件中的文案字符串
- 重点页面：landing、profile、group-detail、posts
- 验收：人工审阅核心用户路径，无禁用词

### P1 — 路由与页面标题治理

目标：URL 与 `<title>` 完全中性化

- 梳理 `router/` 下全部路由定义
- 替换路由 path 中的敏感词（需同步更新内链跳转）
- 更新各页面 `document.title` / SEO meta

### P2 — API 路径与 Swagger 治理

目标：对外接口文档完全中性化

- 梳理 Controller 中全部 `@RequestMapping` 路径
- 新旧路径并行一个版本（301 重定向旧路径），再下线旧路径
- 更新 Swagger `@Operation`、`@Tag`、`@Schema` 注释
- 更新前端 `src/api/` 下的请求路径

### P3 — 数据库历史治理与兼容迁移

目标：逐步清理内部字段命名，减少认知负担

- 为旧字段创建 Flyway 迁移添加别名列或重命名（视影响范围决定）
- 更新 Entity / Repository 命名与旧字段解绑
- 兼容期：Service 层保留映射逻辑，迁移稳定后移除
